登录状态维持机制实现

场景描述

登录系统后，每次点击刷新，都会把当前的登录状态清除，用户体验很差。解决这个问题的方法就是把token存储在localStorage中，登录后即使刷新页面也能自动保持登录状态。未登录时刷新页面虽然和未实现登录状态维持机制看起来没区别，但是机制更加完善，提升了用户体验。

先来理解一下浏览器的存储机制和刷新操作对数据的影响：

  • sessionStorage: 页面刷新后会保留，但关闭浏览器标签页或窗口后会清除
  • localStorage: 页面刷新后保留，关闭浏览器后也保留，只有手动清除或代码清除才会删除
  • 内存中的变量: 页面刷新后会完全清除，重新初始化

原理概述

当用户登录成功后，系统会把**登录凭证（token）**保存在浏览器的localStorage中。每次页面刷新或重新打开时，系统会自动检查localStorage中是否有有效的token，如果有就调用 /me 接口验证token是否还有效，验证通过就自动恢复登录状态，让用户无需重新登录。同时所有的API请求都会自动携带这个token，如果服务器返回401错误（token失效），系统会自动清除本地状态并跳转到登录页面。

核心流程：登录 → 保存token → 页面刷新 → 读取token → 验证token的有效性 → 自动恢复登录状态

总结

前端处理Token过期的核心步骤：

  1. 检测：通过401状态码检测token过期

  2. 清除：删除本地保存的token和用户状态

  3. 提示：告知用户登录已过期

  4. 跳转：引导用户重新登录

关键是要让用户知道发生了什么，并引导他们重新登录。

关键技术点

  • localStorage: 浏览器提供的持久化存储，页面刷新后数据不会丢失，用于保存用户的登录token
  • React Context: React的全局状态管理机制，让所有组件都能访问用户信息，无需层层传递props
  • useMount: 自定义Hook，在组件挂载时执行一次，用于初始化操作
  • useAuth Hook: 自定义Hook，提供认证相关的功能（登录、注册、登出），简化组件中的认证逻辑
  • useHttp Hook: 自定义Hook，自动为所有API请求添加token，统一处理身份验证

核心就是：登录保存token → 页面加载读取token → API请求携带token → 失效自动清除token。
